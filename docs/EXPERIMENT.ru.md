# Экспериментальная методика

Язык: [English](EXPERIMENT.md) | Русский

## Назначение

Этот документ описывает экспериментальную методику, используемую в прототипе PQC CI/CD-конвейера.

Эксперимент сравнивает три TLS-профиля в одном воспроизводимом CI/CD-окружении:

- `classical` — базовый классический профиль TLS 1.3;
- `hybrid` — TLS 1.3 с классическим и постквантовым установлением ключа;
- `pqc` — максимально поддерживаемый профиль с постквантовым установлением ключа.

Цель эксперимента — оценить инженерную интеграцию криптографических профилей в конвейер поставки ПО, а не выполнить benchmark отдельного криптографического примитива.

## Сравниваемые профили

| Профиль | Группа обмена ключами | Тип сертификата | Роль |
|---|---|---|---|
| `classical` | `X25519` | ECDSA / P-256 | Базовый профиль TLS 1.3 |
| `hybrid` | `X25519MLKEM768` | ECDSA / P-256 | Миграционный профиль с классическим и PQC-установлением ключа |
| `pqc` | `MLKEM768` | ECDSA / P-256 | Максимально поддерживаемый профиль с PQC-установлением ключа |

Профиль `pqc` намеренно сохраняет классический сертификат ECDSA. Поэтому он рассматривается как максимально поддерживаемый PQC-профиль, а не как полностью постквантовая TLS/X.509-конфигурация.

## Экспериментальное окружение

Окружение задаётся Docker-конфигурацией и файлами репозитория.

Основные компоненты:

- Docker-образ на основе Ubuntu 24.04;
- OpenSSL 3.5.6, собираемый внутри Docker-образа;
- тестовый стенд Docker Compose;
- `openssl s_server` в роли TLS-сервера;
- `openssl s_client` в роли TLS-клиента;
- Bash-скрипты для оркестрации;
- Python-скрипты для агрегации;
- GitHub Actions для CI-запуска.

## Стадии конвейера

Для каждого профиля конвейер выполняет одну и ту же последовательность:

1. проверяет наличие или выполняет сборку Docker-образов;
2. генерирует временный CA и серверные сертификаты;
3. запускает TLS-сервер;
4. выполняет повторяющиеся TLS-рукопожатия со стороны клиента;
5. записывает сырые CSV-метрики;
6. формирует отчёты в форматах Markdown, CSV и JSON;
7. очищает Docker Compose-окружение.

Параметры конкретных профилей хранятся в `configs/profiles/*.yml`. Логика конвейера остаётся общей для всех профилей.

## Проверка TLS

Каждый профиль ограничивает TLS-согласование ровно одной группой обмена ключами:

- `classical`: `X25519`;
- `hybrid`: `X25519MLKEM768`;
- `pqc`: `MLKEM768`.

Конвейер проверяет:

- успешность TLS-соединения;
- наблюдаемую версию TLS;
- результат проверки сертификата;
- ожидаемую и фактическую группу обмена ключами;
- код завершения OpenSSL;
- вывод ошибки при неуспешном тесте.

Если OpenSSL не выводит согласованный временный ключ, но соединение успешно устанавливается с TLS 1.3, сертификат проходит проверку, а обе стороны ограничены одной настроенной группой, конвейер записывает эту группу с признаком подтверждения:

```text
restricted-single-group
```

## Метрики, отчёты и графики

Сырые метрики TLS-рукопожатий сохраняются в файл:

```text
artifacts/metrics/raw-<profile>.csv
```

Метрики длительности стадий pipeline сохраняются в файл:

```text
artifacts/metrics/stages-<profile>.csv
```

Runtime- и ресурсные метрики сохраняются в файлы:

```text
artifacts/metrics/runtime-<profile>.csv
artifacts/metrics/container-stats-<profile>.csv
```

Отчёты по каждому профилю сохраняются в файлы:

```text
artifacts/reports/summary-<profile>.md
artifacts/reports/summary-<profile>.csv
artifacts/reports/summary-<profile>.json
```

Общие отчёты сохраняются в файлы:

```text
artifacts/reports/summary-all.md
artifacts/reports/summary-all.csv
artifacts/reports/summary-all.json
```

Графики создаются в формате SVG в каталоге:

```text
artifacts/reports/charts/
```

Основные интерпретируемые показатели:

| Показатель                   | Смысл                                                                  |
| ---------------------------- | ---------------------------------------------------------------------- |
| `success_rate`               | Надёжность профиля при повторных TLS-рукопожатиях                      |
| `avg_handshake_ms`           | Среднее время TLS-рукопожатия                                          |
| `median_handshake_ms`        | Медианное время TLS-рукопожатия                                        |
| `total_pipeline_duration_ms` | Общая длительность запуска pipeline для профиля                        |
| `test_duration_ms`           | Длительность стадии интеграционного тестирования                       |
| `server_cpu_max_percent`     | Максимальное сэмплированное потребление CPU контейнером TLS-сервера    |
| `server_memory_max_mib`      | Максимальное сэмплированное потребление памяти контейнером TLS-сервера |
| `dominant_actual_group`      | Наиболее часто наблюдаемая или выведенная группа обмена ключами        |
| `dominant_group_evidence`    | Признак, подтверждающий фактическую группу                             |
| `cert_chain_size_bytes`      | Суммарный размер сертификатов CA и сервера                             |
| `server_key_size_bytes`      | Размер сгенерированного серверного ключа                               |

## Воспроизведение

Сборка образов:

```bash
make build
```

Запуск всех профилей:

```bash
make run-all
```

Полная локальная проверка проекта:

```bash
make check
```

Просмотр возможностей OpenSSL/PQC:

```bash
make crypto-capabilities
```

## Ожидаемый результат

Успешный эксперимент должен сформировать:

* 30 успешных рукопожатий для `classical`;
* 30 успешных рукопожатий для `hybrid`;
* 30 успешных рукопожатий для `pqc`;
* `TLSv1.3` как наблюдаемую версию TLS;
* `X25519`, `X25519MLKEM768` и `MLKEM768` как фактические группы соответствующих профилей;
* отчёты в форматах Markdown, CSV и JSON.

## Ограничения

* Тестовый стенд использует `openssl s_server` и `openssl s_client` вместо полноценного прикладного сервера.
* Профиль `pqc` использует установление ключа ML-KEM, но сохраняет классический сертификат ECDSA.
* Метрики CPU и памяти не собираются как самостоятельные поля отчёта.
* Измеренное время рукопожатия включает выполнение команды и накладные расходы контейнерной сети.
* Прототип предназначен для исследовательского и учебного использования, а не для промышленного криптографического развёртывания.
